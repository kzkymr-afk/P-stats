# 別アプリ：マスター配信ボタン実装指示書

スプレッドシートの GAS メニュー **「マスター配信」→「GitHub Actions で master_out をデプロイ」** と **同じ処理**（`deploy-master-out-pages.yml` の `workflow_dispatch`）を、**別アプリのボタン**から起動するための要件・API・セキュリティ・受け入れ条件をまとめたものです。

全体の前提・トラブルシュートは **`docs/MASTER_OUT_DEPLOY_GAS.md`**、列仕様は **`docs/機種マスター1シート仕様.md`** を参照してください。

---

## 1. このボタンが起こすこと（期待される結果）

1. GitHub Actions **`master_out を GitHub Pages に配信`** が **手動実行相当**で走る。
2. ワークフローが **公開 CSV**（Repository variables の `MASTER_ONE_SHEET_CSV_URL`）を取得し、`scripts/convert_master_one_sheet.py` で **`gh-pages` ブランチの `master_out/`** を更新する。
3. 成功後、アプリ利用者は **GitHub Pages 上の `master_out/index.json` / `master_out/machines/*.json`** が更新される（反映まで **数分**かかることがある）。

**注意:** 更新されるのは **`main` ブランチ上の `master_out/` ではなく**、原則 **`gh-pages` ブランチ**です。確認先を取り違えないこと（`MASTER_OUT_DEPLOY_GAS.md` §4.1）。

---

## 2. サーバー／GAS 側の前提（実装前チェック）

別アプリから呼ぶ前に、次が満たされていること。

| 項目 | 内容 |
|------|------|
| GAS ウェブアプリ | `gas_project/gas.gs` を **ウェブアプリとしてデプロイ**済みで、**POST 可能な URL**（`/exec`）があること。 |
| Script Properties | `GITHUB_PAT` / `GITHUB_REPO` が GAS に設定済み（`MASTER_OUT_DEPLOY_GAS.md` §1）。 |
| Webhook 用シークレット | **`MASTER_DEPLOY_WEBHOOK_SECRET`** を GAS の Script Properties に設定済みであること（未設定だと別アプリからは **`webhook_not_configured`** で拒否）。 |
| GitHub 側 | リポジトリの **Repository variables** に **`MASTER_ONE_SHEET_CSV_URL`** が正しい **ウェブ公開 CSV** を指していること。 |

**動作確認（任意）:** 同じ GAS URL に `mode=ping` を POST し、`master_deploy_webhook: true` が返ることを確認すると安全です（`gas.gs` の `doPost`）。

---

## 3. 推奨アーキテクチャ（セキュリティ）

`MASTER_DEPLOY_WEBHOOK_SECRET` は **クライアントアプリに埋め込まない**ことを強く推奨します。

| 方式 | 推奨度 | 説明 |
|------|--------|------|
| **A. 自前バックエンド（プロキシ）** | 高 | 別アプリ → あなたの API（認証済み）→ GAS `POST`（シークレットはサーバー環境変数）。 |
| **B. モバイルのみ + Keychain 等** | 中 | シークレットを端末の安全領域に保存。漏洩リスクは残る。 |
| **C. GitHub API を別アプリから直接叩く** | 別途 | PAT をサーバー側に置き `workflow_dispatch` を呼ぶ。GAS を経由しない。本書のスコープ外だが可能。 |

本書は **GAS 経由（`MASTER_OUT_DEPLOY_GAS.md` §3 と同一）** を前提にします。

## 4. HTTP API（GAS `doPost`）

### 4.1 エンドポイント

- GAS エディタの **デプロイ** から取得した **ウェブアプリ URL**（例: `https://script.google.com/macros/s/＜ID＞/exec`）
- **メソッド:** `POST`
- **Content-Type:** `application/x-www-form-urlencoded` を推奨（既存 `doPost` と整合）

リダイレクトが挟まるクライアントでは **`curl -L`** 相当の追従、または最終 URL を取得する実装が必要な場合があります（`MASTER_OUT_DEPLOY_GAS.md` の curl 例参照）。

### 4.2 パラメータ（マスター配信）

| 名前 | 必須 | 値 |
|------|------|-----|
| `mode` | はい | 固定: `trigger_master_out_deploy` |
| `secret` または `deploy_secret` | はい | GAS Script Properties の **`MASTER_DEPLOY_WEBHOOK_SECRET` と完全一致** |

### 4.3 レスポンス（JSON）

成功例:

```json
{ "ok": true, "github_status": 204 }
```

失敗例（例）:

```json
{ "ok": false, "error": "forbidden" }
```

```json
{ "ok": false, "error": "webhook_not_configured" }
```

```json
{ "ok": false, "github_status": 401, "github_body": "..." }
```

`github_status` は GitHub REST API（workflow dispatch）の HTTP ステータスに相当します。**204** は多くの場合「ディスパッチ受理」。

### 4.4 curl 例

```bash
curl -sS -L -X POST "https://script.google.com/macros/s/＜デプロイID＞/exec" \
  -d "mode=trigger_master_out_deploy" \
  -d "secret=＜MASTER_DEPLOY_WEBHOOK_SECRET＞"
```

## 5. 別アプリ UI／UX 要件（ボタン実装）

### 5.1 ボタン押下前

- 文言例: 「スプレッドシートを保存し、**公開 CSV が反映されるまで数十秒〜1分**空けてから実行してください」（`MASTER_OUT_DEPLOY_GAS.md` 前提）。
- （任意）**確認ダイアログ**: 「GitHub Actions で master_out を再配信します。よろしいですか？」

### 5.2 実行中

- ボタンを **無効化**、インジケータ表示。
- **タイムアウト**を設ける（例: 30〜60 秒）。GAS → GitHub までの往復で遅延しうる。

### 5.3 成功時

- `ok: true` を表示し、**GitHub Actions の該当ワークフロー一覧**へのリンクを出すと運用が楽です（リポジトリ URL は固定または設定画面で入力）。
  - 例: `https://github.com/<owner>/<repo>/actions/workflows/deploy-master-out-pages.yml`
- 「**数分後**に Pages の `master_out` が更新されます」と注記。

### 5.4 失敗時

- `error` / `github_status` / `github_body`（先頭数百文字）を **開発者向けログ**に残し、ユーザーには短いメッセージ（例: 「配信の開始に失敗しました。設定を確認してください」）。
- 典型原因は `MASTER_OUT_DEPLOY_GAS.md` §4 の表を参照。

### 5.5 CORS

- **ブラウザの JavaScript から GAS URL に直接 POST** すると **CORS で失敗しやすい**です。Web 版は **自前バックエンド経由**を推奨（`MASTER_OUT_DEPLOY_GAS.md` §3「別アプリ実装時の注意」）。

## 6. 受け入れ条件（Done の定義）

- [ ] ボタン操作で GAS に POST し、`{ "ok": true, ... }` が返る（または同等の成功判定）。
- [ ] GitHub の Actions で **`master_out を GitHub Pages に配信`** が **新規実行**される。
- [ ] 実行が **緑**で完了し、`gh-pages` の `master_out/` に期待する更新が入る（確認はブランチ **`gh-pages`**）。
