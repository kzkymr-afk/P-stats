# マスターデータ編集アプリ用：Collector → GAS → スプレッドシート連携 実装指示書

別開発の「マスターデータ編集アプリ」に、**機種ページHTMLを取得 → GAS に送信 → スプレッドシート（30列 A〜AD）が更新される**一連の動作を実装するための指示書です。別チャットや別担当者に渡して実装してもらう想定です。

---

## 1. 全体の流れ

```
[アプリ] 機種ページのHTMLを取得
    ↓
[アプリ] GAS の Web アプリ URL に POST（text=HTML, id=機種ID）
    ↓
[GAS] doPost で受信 → processPachinkoText で HTML を解析
    ↓
[GAS] スプレッドシートの 2 行目に 1 行挿入（30列 A〜AD 形式）
    ↓
[GAS] 応答 "成功: 機種名" を返す
    ↓
[アプリ] 応答を表示 or ローカルで「完了」扱いにする
```

- **スプレッドシート**: 1 行目＝ヘッダー（30列 A〜AD）、2 行目以降＝1 機種 1 行のデータ。GAS は「2 行目の前に 1 行挿入」する。
- **GAS**: デプロイ済み Web アプリ（doPost）が入口。同一の GAS を P-stats の Pachi_Collector とマスタ編集アプリの両方から利用する想定です。

---

## 2. GAS API 仕様（アプリ側が呼び出すインターフェース）

### 2.1 エンドポイント

- **URL**: Web アプリとしてデプロイした GAS の URL（例: `https://script.google.com/macros/s/xxxxx/exec`）。  
  ※ 実際の URL は GAS プロジェクトの「デプロイ」で取得。P-stats では `scraper/Pachi_Collector.py` の `GAS_URL` を参照。
- **メソッド**: **POST**
- **Content-Type**: `application/x-www-form-urlencoded`（フォームパラメータで送る）

### 2.2 パラメータ

| パラメータ | 必須 | 説明 |
|------------|------|------|
| `mode` | 条件付き | `get_done_list` のときはこれだけ送る（下記「完了済み一覧取得」）。 |
| `id` または `url` | 解析時 | 機種ID（数値文字列）。HTML に機種IDが無い場合のフォールバック。 |
| `text` | 解析時 | 機種詳細ページの **HTML 文字列**（生HTML or 前処理済み）。 |

- **解析して 1 行追加したいとき**: `text` と `id`（または `url`）を送る。`mode` は送らない。
- **完了済み機種ID一覧が欲しいとき**: `mode=get_done_list` だけ送る。`text` は不要。

### 2.3 レスポンス（テキスト）

- **解析成功時**: `"成功: 機種名"`（機種名は解析結果）
- **解析失敗時**: `"解析エラー: メッセージ"`
- **完了済み一覧取得時**: 機種IDをカンマ区切りで返す（例: `"123,456,789"`）。該当なしやヘッダーだけのときは空文字や短い文字列の可能性あり。
- **その他エラー**: `"Error: No data"` や `"System Error: ..."` など

## 3. アプリ側で実装する動作

### 3.1 機種 1 件をスプレッドシートに追加する流れ

1. **機種IDを決める**  
   例: 一覧から選択した機種の ID、または詳細 URL の末尾の数値。
2. **その機種の詳細ページ HTML を取得する**  
   - ブラウザで開いているページの HTML をそのまま渡す、  
   - またはアプリ内で `https://p-town.dmm.com/machines/{機種ID}` 等に HTTP アクセスして HTML を取得する。
3. **（推奨）送信前に HTML を軽く前処理する**  
   - `<script>...</script>` と `<style>...</style>` を削除すると GAS の解析が安定しやすい。  
   - 「ゲームフロー」より後を削除、「ユーザー口コミ・評価詳細」より後を削除すると、解析対象を機種基本情報に絞れる。  
   ※ 既存実装: `P-stats/scraper/Pachi_Collector.py` の 149–155 行付近を参照。
4. **GAS に POST する**  
   - ボディ: `text={前処理したHTML}&id={機種ID}`（または `url={URL}` で機種IDが含まれる形でも可）。  
   - 実際の GAS の URL は、マスタ編集アプリの設定画面で指定できるようにしておくとよい。
5. **レスポンスを解釈する**  
   - `"成功:"` または `"完了:"` を含む → 成功。必要なら「この機種IDは登録済み」などローカルに記録。  
   - 上記以外 → エラーとしてメッセージを表示。

### 3.2 （任意）完了済み機種ID一覧を取得する

- 重複登録を避けたい場合などに利用。
- **POST**: `mode=get_done_list` のみ（他パラメータ不要）。
- **レスポンス**: スプレッドシートで「ステータス」列（I列）が「完了」の行の「機種ID」（B列）を、カンマ区切りで返す。
- アプリ側では、この文字列を `split(',')` して Set 等で保持し、一覧に表示する機種のうち「すでにスプレッドシートに登録済み」を判定する。

## 4. 30列マスター（スプレッドシート）の列定義（A〜AD）

GAS は **1 行目をこのヘッダー** として扱い、2 行目に 1 行挿入します。マスタ編集アプリで同じスプレッドシートを編集する場合も、列順・列名を揃えてください。

| # | 列 | 列名 | GAS が自動入力するか | 説明 |
|---|-----|------|----------------------|------|
| 1 | A | 導入開始日 | ○ | 「導入開始日」を YYYY/MM/DD に変換。 |
| 2 | B | 機種ID | ○ | URL やパラメータ `id` から取得。 |
| 3 | C | 機種名 | ○ | HTML の `<h1>` から取得。 |
| 4 | D | メーカー | ○ | 「メーカー名」「メーカー」周辺の td から取得。 |
| 5 | E | 確率 | ○ | 「大当り確率」の 1/XX.X 形式。 |
| 6 | F | 機種タイプ | ○ | スペックから判定。`st` / `kakugen` / `羽根モノ` / `1種2種混合機` など。 |
| 7 | G | スペック | ○ | 確率分母などから判定。`羽根モノ` / `ミドル` / `ライトミドル` / `甘デジ` / `その他`。 |
| 8 | H | 特徴タグ | ○ | アイコン等から。`ST` / `遊タイム` / `c時短` / `設定付` などを `/` 区切り。なしのとき「なし」。 |
| 9 | I | ステータス | ○ | 必須項目が揃っていれば「完了」、不足なら「要確認」。 |
| 10 | J | 更新対象 | ○（新規行は「対象」） | `対象` / 空＝master_out 配信対象、`対象外`＝配信しない（既存 JSON 保持）。詳細は `docs/機種マスター1シート仕様.md`。 |
| 11〜18 | K〜R | モード0 〜 モード7 | 空欄 | ユーザー入力用。 |
| 19〜30 | S〜AH | 当たり1 〜 当たり12 | 空欄 | ユーザー入力用。「当たり名/基本出玉/ユニット出玉/最大連結数/滞在モード番号/移行先モード番号」形式。ユニット出玉が0なら完結型。 |

## 5. HTML 解析の前提（GAS 側の仕様）

GAS は次のような HTML を前提に解析しています。アプリから送る HTML も同じソース（p-town 機種詳細ページ）であればそのまま送って問題ありません。

- **解析対象範囲**: 送信された文字列のうち、「ユーザー口コミ・評価詳細」より前かつ「ゲームフロー」より前。
- **導入開始日**: 「導入開始日」に続く「YYYY年M月D日」形式を YYYY/MM/DD に変換。
- **機種ID**: パラメータ `id`（または `url` から数値抽出）。無ければ HTML 内の `/machines/(数値)` から取得。
- **機種名**: `<h1>...</h1>` のテキスト（1 行目・括弧前まで、全角→半角変換あり）。
- **メーカー**: 「メーカー名」「メーカー」に続く `<td>` など。
- **確率**: 「大当り確率」に続く「数字/数字」形式。
- **機種タイプ**: スペックエリアのアイコン alt 等（ST機 → `st`、それ以外デジパチ系 → `kakugen` など）。
- **スペック**: 確率分母から `羽根モノ` / `ミドル` / `ライトミドル` / `甘デジ` / `その他` を判定。
- **特徴タグ**: アイコン・表から ST/遊タイム/c時短/設定付 等を検出し、`/` 区切りで出力。

詳細な正規表現・ロジックは **P-stats リポジトリの `gas_project/gas.gs` の `processPachinkoText`** をそのまま参照してください。

## 7. マスタ編集アプリでの実装チェックリスト（案）

- [ ] 設定画面などで「GAS の URL」を保存できるようにする。
- [ ] 機種詳細の HTML を取得する手段を用意する（URL を開いて HTML を取る、またはユーザーが貼り付けた HTML を使う）。
- [ ] 送信前に HTML の前処理（script/style 削除、ゲームフロー以降の削除など）を行う（任意だが推奨）。
- [ ] `POST {GAS_URL}` に `text` と `id` を送り、レスポンスが「成功:」を含むかで成功/失敗を判定する。
- [ ] （任意）`mode=get_done_list` で完了済み機種ID一覧を取得し、重複登録防止や一覧の「登録済み」表示に使う。
- [ ] エラー時（「解析エラー」「System Error」等）はレスポンス文字列をそのまま or 要約してユーザーに表示する。

## 8. 注意事項

- GAS の Web アプリは「全員がアクセス可」など、匿名で POST できるデプロイにしておく必要があります（マスタ編集アプリからは認証しない想定のため）。
- スプレッドシートの 1 行目は必ず 30 列（A〜AD）のヘッダーにしておく必要があります。GAS は 2 行目に挿入するため、既存データは 2 行目以降に並びます。
- 同一の GAS・同一のスプレッドシートを、P-stats の Pachi_Collector とマスタ編集アプリの両方から使う場合、列構成や `get_done_list` の列番号（機種ID＝B列＝列2、ステータス＝I列＝列9）は **gas.gs の現行仕様と一致させてください**。

以上を前提に、マスタ編集アプリ側で「HTML 取得 → GAS 送信 → スプレッドシート更新」の一連の動作を実装すれば、Pachi_Collector と同じ流れで 30 列（A〜AD）マスターが更新されます。
