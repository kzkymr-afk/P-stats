# エージェント指示書：機種情報取得・クレンジングのハイブリッド化

## 1. 目的

- DMMぱちタウンの機種詳細ページから取得したテキスト（HTML）を GAS で受け取り、**ルールベース**と**LLM（Gemini）**を組み合わせて必要な情報だけを整理し、スプレッドシート（マスターデータ）に保存する。
- 表記揺れの多い「大当たり概要」は LLM で構造化し、それ以外は既存の正規表現を維持・改善する。

---

## 2. 対象ファイル・リソース

| 対象 | パス・場所 | 役割 |
|------|------------|------|
| GAS スクリプト | `gas_project/gas.gs` | HTML 受信、ルール抽出、LLM 呼び出し、検証、スプレッドシート書き込み |
| Python クローラー | `scraper/Pachi_Collector.py` | 機種ページの HTML 取得。必要なら「ゲームフロー」より前だけ送るよう変更 |
| スプレッドシート | ID `1fSGx5EmcSOD68itgBRxjGyUGz0Wh5u1Lnbw-dyvchz4` | マスターデータの保存先。既存の列構成を尊重しつつ、必要なら列を追加 |

---

## 3. 抽出する情報一覧

### 3.1 ルール（正規表現）で抽出する項目

- 導入日
- 機種名
- メーカー名
- 通常時の大当たり確率
- ボーダー（等価ベース）

※既存の `processPachinkoText` 内の正規表現をベースにし、解析対象を「ゲームフロー」より前のテキストに限定して使う。

### 3.2 LLM（Gemini）で抽出する項目（大当たり概要）

- 各大当たり種類ごとに：
  - 出玉数（玉数）
  - R数（回数）
  - 終了後の電サポ回数
  - 終了後のモード（通常 / RUSH / 上位RUSH（LT）など）
- 可能であれば：
  - ボーナス名（「〇〇ボーナス」など）
  - モード名（「〇〇RUSH」「〇〇ゾーン」など）

※入力は「大当たり出玉」「電サポ」周辺の平文テキストブロックに限定する。

### 3.3 その他（既存ロジックを維持）

- 機種タイプ（ST/確変など）、スペック種別、特徴タグ（ST/LT/遊タイム等）、機種ID、更新ステータス は、既存のルール・alt 判定を崩さず維持する。

---

## 4. 実装すべき処理フロー（GAS）

1. **入力の前処理**
   - 受け取った HTML から「ゲームフロー」より前の部分だけを切り出し、解析対象とする（バナー・広告を除外）。
   - 必要に応じて script/style タグ除去など既存のクレンジングを適用。

2. **ルール抽出**
   - 上記テキストに対して、既存の正規表現で以下を抽出する。
     - 導入日、機種名、メーカー、大当り確率、等価ボーダー、大当たり出玉（概要用テキスト）、電サポ回数 など。
   - 既存の `processPachinkoText` のロジックを流用し、解析対象を「ゲームフロー前」に限定した変数（例: `parseTarget`）に対して実行する。

3. **LLM 用テキストの切り出し**
   - 「大当たり出玉」「電サポ」周辺のブロックをタグ除去して平文にし、Gemini に渡すテキストを用意する。

4. **Gemini API の呼び出し**
   - スクリプトプロパティの `GEMINI_API_KEY` を参照する（既存の `getGeminiApiKey()` があれば利用）。
   - プロンプトの要件：
     - 上記テキストから「大当たりの種類ごとの 出玉数・R数・電サポ回数・終了後モード」を抽出する。
     - 可能ならボーナス名・モード名も抽出する。
     - **出力は JSON のみ**とし、キー名を固定する（例: `rounds`, `balls`, `support`, `afterMode`, `bonusName`, `modeName` など）。
   - モデルは Gemini 1.5 Flash または Pro など、GAS から呼びやすいものを指定する。

5. **LLM 結果の検証**
   - 返答をパースし、必須キーが含まれるか、数値が妥当な範囲かなどを簡易チェックする。
   - 検証失敗時は、その大当たり概要部分は空または「要確認」とし、スプレッドシートの「更新ステータス」または専用列に「要確認」を立てる。

6. **スプレッドシートへの書き込み**
   - ルールで得た値と LLM で得た大当たり概要をマージする。
   - 既存の列並び（A: 導入日 ～ L: 更新ステータス）を維持する。
   - 大当たり詳細を保存する列を追加する場合は、既存の `HEADER_ROW` と `newRow` の並びを確認し、列を追加するなら末尾または規定の位置に「大当たり詳細（JSON）」などを追加する。

---

## 5. Pachi_Collector.py での対応（任意だが推奨）

- GAS に送る HTML を、**「ゲームフロー」より前の部分だけ**にすると、バナー誤検知が減る。
- 送信前に `raw_html`（または `clean_html`）から `split("ゲームフロー")[0]` を取った文字列を GAS に送るように変更する。

---

## 6. 制約・注意事項

- **既存の動作を壊さない**
  - `processPachinkoText` の既存の列順（A～L）、`machineType` / `specType` の判定、alt による ST/LT 判定は維持する。
- **フォールバック**
  - Gemini の呼び出し失敗・タイムアウト・パース失敗時は、LLM 部分は空または「要確認」とし、ルールで取れた項目だけで行を書き込む。
- **API キー**
  - Gemini API キーは GAS の「スクリプトのプロパティ」に `GEMINI_API_KEY` で設定されている前提。未設定時は LLM ステップをスキップし、ルールのみで書き込む。
