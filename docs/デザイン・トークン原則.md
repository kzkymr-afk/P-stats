# デザイン・トークン原則（色・不透明度・線幅）

## 結論：直書きはまだ「ゼロ」ではない

`P-stats/Design/` **以外**の Swift にも、`Color.white.opacity(0.xx)` / `Color.black.opacity(0.xx)` / `Color(red: 数字, …)` のような **数値直書き**が残っています。  
移行は進行中であり、**新規・改修では増やさない**ことを最優先にします。

確認用（リポジトリルート）:

```bash
./scripts/check_design_literals.sh
```

- **exit 0** … 上記パターンの数値直書きが Design 外にない  
- **exit 1** … 該当行が列挙される（現状は未完了のため多く出る想定）

## 原理原則（今後の改修・追加で守ること）

1. **数値は View に書かない**  
   不透明度・線幅・角丸の「意味のある単位」は **`DesignTokens`**（`Surface` / `Thickness` / `Elevation` など）か、既存の **`AppGlassStyle`・`ApplicationTheme`・`DesignTokens.ShareCard`** など **Design 配下またはテーマ経由**に置く。

2. **意味を名前で残す**  
   「0.12 の白」ではなく、`elevated` / `strokeInputWell` / `playToolbarBackground` のように **文脈が読める名前**で定義する。演出専用は `Surface.Play` のような名前空間に閉じる。

3. **システム色・テーマ色を優先**  
   意味が「ラベル」「区切り」「プレースホルダ」なら、可能な範囲で **`themeManager.currentTheme` のセマンティック色**や `Semantic` 系に寄せる。

4. **例外は局所化し理由を書く**  
   共有画像のように **画面外レンダリングで独自パレットが必須**な場合は、`DesignTokens` の専用 enum（例: `ShareCard`）に集約し、View 側は **定数名参照のみ**にする。やむを得ない一行だけの場合は **その行の上に一行コメント**で理由を残す。

5. **プレビューは `ThemePreview` で包む**  
   `ThemeManager` を要求する View の `#Preview` は `P-stats/Design/ThemePreviewContainer.swift` の **`ThemePreview { … }`** でラップし、`environmentObject` 欠落によるクラッシュを防ぐ。

6. **マージ前のセルフチェック**  
   関連変更を入れたら `./scripts/check_design_literals.sh` を実行し、**自分が触った範囲で新規の数値直書きが増えていないか**を確認する（プロジェクト全体が green になるまで必須とはしないが、**悪化させない**）。

## 定義の置き場所（参照順の目安）

| 種類 | 主な置き場 |
|------|------------|
| 数値トークン（OS 非依存） | `P-stats/Design/DesignTokens.swift` |
| ガラス・グラデ・背景 | `AppGlassStyle` 等 `Design/` |
| スキンごとの色 | `ApplicationTheme` 実装・`ThemeManager` |
| 共有画像専用 | `DesignTokens.ShareCard` と `SessionShareTemplates` 周辺のヘルパ |
