# システム改修指示書：パチンコあたり入力フローの「ユニット連結型（積み上げ）」への変更

## 1. 目的（UXゴール）

- **ゴール**: 「後出し上乗せ機（例: カバネリ2）」に対し、ユーザーがあとから出玉を修正するストレスをなくし、**演出の告知タイミングに合わせて出玉をタップして積み上げ**できる入力フローを実現する。
- **現行の課題**: 「1つのあたり＝固定総出玉（完結型）」だと、上乗せ演出で最終出玉が後出しされる機種で入力ミス・修正が発生しやすい。
- **方針**: 当たりを「基本出玉＋（任意）ユニット出玉×回数」の**階層構造**として扱い、履歴確定はユーザーの明示操作（確定/次へ）で行う。

---

## 2. データ仕様（スプレッドシート：当たりセルの拡張）

### 2.1 対象

- 機種マスターの「当たり1〜当たり10」セル（現状スラッシュ区切りの文字列）。

### 2.2 新フォーマット（ユニット連結型）

当たりセルのデータ格納ルールを以下に拡張する。

**新形式（6要素）**

`当たり名 / 基本出玉 / ユニット出玉 / 最大連結数 / 滞在モードID / 移行先モードID`

- **当たり名**: 表示名
- **基本出玉**: 初回タップで即計上される出玉量
- **ユニット出玉**: 上乗せ（追撃）1回あたりの加算量（0なら上乗せなし）
- **最大連結数**: 上乗せボタンを表示し続ける最大回数（例: 5 なら最大5回まで）
- **滞在モードID**: その当たりが発生するモード（0〜7 を推奨）
- **移行先モードID**: 当たり後に遷移するモード（0〜7 を推奨）

### 2.3 後方互換（旧フォーマットの扱い）

既存データ・既存入力との互換のため、GAS/アプリ側のパースは以下のルールで両対応にする。

**旧形式（4要素）**

`当たり名 / 出球数 / 滞在モードID / 移行先モードID`

- 新形式へ読み替え: `基本出玉=出球数`, `ユニット出玉=0`, `最大連結数=1`

**旧形式（5要素：電サポ回数あり）**

`当たり名 / 出球数 / 滞在モードID / 移行先モードID / 電サポ回数`

- 新形式へ読み替え: `基本出玉=出球数`, `ユニット出玉=0`, `最大連結数=1`
- **電サポ回数は現行仕様どおり保持したい場合**は、新形式に拡張するか（例: 7要素目）/ 別フィールドへ移すか、プロジェクト側で決定すること（本指示書では必須ではない）。

### 2.4 入力例（カバネリ2想定）

- 初当り（海門決戦へ）  
  `海門決戦BONUS/750/0/1/1/2`
- 右打ち（カバネリBONUS：後出し上乗せ）  
  `カバネリBONUS/1500/1500/5/3/3`

解説：最初に1500個（基本）を計上し、以降「+1500（追撃）」を最大5回まで積み上げられる。

## 3. GAS 改修指示（マスター → JSON 生成側）

> **注意**: `gas_project/gas.gs` は現状「HTML解析→シートへ追記」が主で、JSON配信は別系統（Python変換等）に寄っている。  
> 本改修の要点は「当たりセル文字列のパース結果（base/unit/max）をアプリに渡せる形にする」こと。

### 3.1 パース関数（例）

GAS（またはアプリ・変換スクリプト）に、当たりセルのパース関数を実装する。**旧4要素/旧5要素/新6要素**を許容し、欠損はデフォルト補完する。

#### 期待する出力（共通オブジェクト）

- `name`: string
- `baseOut`: number
- `unitOut`: number
- `maxStack`: number
- `currentMode`: number
- `nextMode`: number

（任意で `densapo` 等を追加）

### 3.2 JSON 生成（アプリへ渡す形）

アプリ側は「当たり一覧」を以下のように扱える必要がある。

- **モードごとに候補ボタンを出す**ため、`currentMode` でフィルタできる
- **ユニット積み上げ**に必要な `baseOut/unitOut/maxStack` が入っている

例（概念）

```json
{
  "mode_id": 3,
  "name": "RUSH",
  "bonuses": [
    { "name": "カバネリBONUS", "baseOut": 1500, "unitOut": 1500, "maxStack": 5, "next_mode_id": 3 }
  ]
}
```

## 4. アプリ改修指示（入力UI/UXと状態遷移）

### 4.1 入力UI（基本）

- **当たりボタン（メイン）**: `当たり名（基本出玉）` を表示し、タップで「今回の出玉」を開始。
- **追撃ボタン（サブ）**: `unitOut > 0` のときに表示（例: `+1500（追撃）`）。

### 4.2 状態（セッション）モデル

1回の「当たり入力セッション」で保持する最小状態は以下。

- `selectedBonusId`（または name）
- `baseOut`
- `unitOut`
- `maxStack`
- `unitCount`（押下回数、初期0）
- `currentTotal = baseOut + unitOut * unitCount`

### 4.3 操作フロー（確定タイミング）

- **開始**: 当たりボタンをタップ → `baseOut` を即時計上して表示（unitCount=0）
- **積み上げ**: 追撃ボタンを押すたびに `unitCount++`、合計表示更新
- **上限**: `unitCount >= maxStack` で追撃ボタンを無効化/非表示
- **確定**: 「次の回転へ」または「確定」操作で、その時点の `currentTotal` を**1履歴として確定**する  
  - これにより「後出し上乗せ」の追加タップが完了したタイミングで確定できる

### 4.4 取消・やり直し（推奨）

- **1つ戻す**: 追撃を1回分取り消し（`unitCount--`）できる導線
- **当たり選択のやり直し**: セッションを破棄して当たり選択画面へ戻る

---

## 5. 移行・運用ルール

### 5.1 既存マスターの移行

- 既存の当たりセル（4要素/5要素）は **そのままでも動く**（unitOut=0, maxStack=1 扱い）。
- 後出し上乗せ機のみ、新6要素で入力する運用に段階移行できる。

### 5.2 バリデーション（最低限）

- 数値項目（base/unit/max, mode）は `Number()` で解釈し、不正は 0 や 1 にフォールバック
- `maxStack` は 1 以上（0以下は 1 扱い）
- `unitOut==0` の場合、追撃ボタンを出さない

---

## 6. 期待される効果

- **入力ミスの撲滅**: 「最終出玉が後出し」の機種でも「出た分だけ足す」方式で誤りが起きにくい。
- **実機シンクロ**: 追撃演出のタイミングでタップするため記録が遊技体験の一部になり、ストレスが減る。
- **共存**: 完結型（unit=0）と上乗せ型（unit>0）を同一データ構造で扱える。
